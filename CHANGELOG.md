# Changelog

## 2026-07-04 — Iteration 0: concept and creative direction

The plan for this iteration was to turn the founding idea, a native macOS app that tracks digital life in bytes, into a grounded concept brief before any code is written.

Progress: a 14-agent workflow ran five macOS feasibility researchers (AI tokens, network, disk, screen time, input), one competitive landscape scan, five independent creative directions, and a three-judge panel scoring desirability, conceptual honesty, and buildability. The Ledger direction won (scores 8, 9, 8) and was synthesized with grafts from the other directions into README.md. Raw research was preserved under docs/research/. Next step: the founder picks the creative direction and distribution channel, then iteration 1 plans the v1 architecture.

## 2026-07-04 — Iteration 1: implementation plan and handoff to the dev Mac

The plan for this iteration was to design the full v1 scaffold and hand the project off through GitHub, since development happens on a different Mac.

Progress: decisions were recorded (menubar-first form factor, Developer ID distribution, core-plus-collectors architecture, Swift Package with zero dependencies). An architect agent produced the detailed implementation design, including an empirically verified dedup key for Claude Code token parsing, and it was saved as PLAN.md. The repo was initialized in git and pushed to github.com/ViGeng/byteslife. Next step: execute PLAN.md on the development Mac, starting with the Package.swift skeleton and `swift build`.

## 2026-07-07 — Iteration 2: v1 scaffold executed, first running demo

The plan for this iteration was to execute PLAN.md top to bottom on the development Mac and produce the first runnable minimum version.

Progress: a six-stage sequential workflow implemented the package skeleton, the model layer, the WAL SQLite SampleStore with UPSERT minute buckets, the CounterAccumulator, the collector framework, the AI chain (Claude Code parser with the verified dedup key, resumable FileTailer, vnode-watcher source), the four system collectors, the MenuBarExtra panel, and scripts/package-app.sh. A five-dimension review with two-vote adversarial verification then confirmed three real defects: a transcript-watcher file-descriptor leak, a non-atomic AI ingest that could silently lose tokens, and an input-tap failure mode that reported running while counting nothing. A fix pass resolved all three plus the highest-value minors, and its own re-verification caught a deadlock the tap fix had introduced, which was then removed. Ground truth on this machine: swift build, swift build -c release, and swift test (81 tests) are all green, and dist/ByteLife.app launches and runs. On first launch the AI collector backfilled 29 days of real token history from local Claude Code transcripts (about 2.1M input, 9.2M output, 986M cache-read, and 37M cache-creation tokens across 5,001 deduplicated events) while the network, disk, and screen collectors recorded live data. Input counting correctly sits in the needs-permission state until Input Monitoring is granted from the panel. Next step: grant Input Monitoring, let the collectors run for a few days to validate the numbers, then start the Ledger skin and the Reconcile ritual on top of the scaffold.

## 2026-07-07 — Iteration 3: the Ledger experience, version 0.2.0

The plan for this iteration was to apply the Ledger creative direction on top of the scaffold: the day-sheet skin, the Reconcile ritual with the immutable receipt, and the General Ledger window, per the plan section appended to PLAN.md.

Progress: a three-stage workflow built the ledger domain layer (five accounts with the concept sheet's exact names and debit and credit sides, cache tokens on a memo line, the running balance as posted byte volume, an additive schema-v2 reconciliations table, the deterministic SHA-256-stamped ReceiptComposer, and the MarginNotes rule engine), then the panel skin in the ledger palette with monospaced aligned columns, then the Reconcile ritual with the thermal-strip receipt and the General Ledger window with periods, day detail, and the all-history trial balance. The review pass confirmed three findings, all fixed: past-day closes now stamp POSTED IN ARREARS with a disclosure instead of baking today's collector states into the immutable artifact, brass gold is gated strictly to BALANCED everywhere, and a posted day now propagates to an already-open window via a notification. Founder feedback added during the iteration: the panel is now dynamic (immediate refresh on open, figures rolling with a numeric transition, two-second live ticks while open, thirty-second label-only cadence while closed), and the multi-device direction was designed and recorded in docs/design/multi-device.md (holdings with stable identities, additive merge for device-own metrics, source-attributed dedup for shared sources, and agentless SSH collection for remote machines). Ground truth: 137 tests green, the live database migrated in place from v1 to v2 with all rows preserved (verified by count before and after, with a one-time backup kept in the data directory), and ByteLife.app 0.2.0 is running in the menubar. Next step: run the Ledger for a few days, close the first real books nightly, then take up the weekly Statement, the schema-v3 holdings groundwork, or the app rename, whichever the founder prioritizes.

## 2026-07-07 — Iteration 4: the live dashboard panel, version 0.3.0

The plan for this iteration was to replace the ledger-styled panel with a dashboard, per founder direction, keeping the Ledger as the record layer.

Progress: the design went through three founder-driven passes. A workflow first built the design-agnostic meter core (an indexed midnight-crossing minuteSeries store query and a pure MeterBridge model with EMA-smoothed live rates from successive 2-second snapshots, floored normalization, session peak-hold, and honest UNCALIBRATED tags) plus an Instrument-style panel. Founder feedback then asked for a fancier byte-native look and finally for real charts, so the shipped panel is the Byte Flow deck, designed and implemented directly in the main session: a dark byte-chassis with a hero flow chart of the last thirty minutes (network teal and disk violet on one shared absolute scale, Swift Charts), a gradient sparkline per channel in its own signal color with the window maximum held as a dial-white dot, glowing live readouts and a pulsing live dot gated on real liveness thresholds, an attention ring for the one absolute-scale channel, a prompted-versus-generated token ratio bar, a LIVE chip, and a hex ticker printing the last four real inter-poll byte deltas. The review pass confirmed and fixed a phantom-rate defect (stale EMA state across panel close and reopen showed decaying rates with no traffic and could forge a session peak from a close-gap average; reopen now cold-starts rates while keeping peaks), and the EMA now snaps to true zero below the liveness threshold so live lights actually turn off. Ground truth: 166 tests green, and ByteLife.app 0.3.0 is running with the deck live. Next step: let the deck run through real workdays, grant Input Monitoring to light MECHANICS, and pick the next layer (weekly Statement, holdings schema groundwork, or the app rename).

## 2026-07-07 — Iteration 5: adaptive appearance, modern ledger window, shareable receipt, version 0.4.0

The plan for this iteration was the polish pass from founder feedback on 0.3.0: make the deck appearance-adaptive, bring the General Ledger window up to the panel's modern style, and turn the receipt into a real, shareable thermal artifact.

Progress: a three-stage workflow executed the plan. LatticePalette became a function of the color scheme (dark keeps the shipped values exactly; light uses pinned variants with deeper channel colors and half-strength glows), and the panel, charts, and window now track the system appearance with nothing hard-coded dark. The receipt kept its verbatim stored text but now renders as a genuine thermal strip: torn sawtooth edges, paper color in both appearances with a lift shadow, and a footer barcode drawn deterministically from the receipt's SHA-256 content hash with the hash printed beneath, so the tamper-evidence is the visual signature. Both receipt surfaces gained Share (system share sheet with a 2x PNG) and Save (PNG or vector PDF through a save panel); the export paths were verified working headless, not assumed. The review pass confirmed zero findings; from its minors, the light-mode receipt gained a faint edge stroke and a failed save now raises an alert instead of silently discarding the write. Ground truth: 174 tests green and ByteLife.app 0.4.0 running. Next step: live with 0.4.0 across appearance switches and real reconciles, then pick the next layer (weekly Statement, the schema-v3 holdings groundwork from docs/design/multi-device.md, or the app rename).

## 2026-07-07 — Iteration 6: working share and the Back Office redesign, version 0.5.0

The plan for this iteration came from founder feedback on 0.4.0: the receipt share attached nothing (Messages activated with no payload), and the General Ledger window needed a genuine redesign rather than another restyle. The receipt paper itself was approved and stays untouched.

Progress: the share bug's root cause was the deferred Transferable render — the share service pulled the PNG after the menubar panel had closed and the transfer context was gone. Sharing now renders eagerly on click, writes the PNG to a per-invocation unique temp path, and presents an NSSharingServicePicker anchored to a live view; the picker is retained until its delegate reports close, and both render and anchor failures surface alerts instead of silent no-ops. The window became the Back Office: the accounts rail and the near-empty history chart card are gone, the periods sidebar now carries per-day amber token and teal byte activity minis (the list itself is the history chart) with a pinned all-time summary card, and the main pane is a day dashboard — full date and stamp, posted-volume hero, a 24-hour teal/violet flow chart of the selected day, a Token card plus a two-by-two account grid each with 24 hourly bars in its channel color and debit/credit figures in new scheme-aware colors, and the receipt strip with the share toolbar below for posted days. The data comes from a new indexed single-day hourly query and tested pure models (DayStory, DayActivity, DayLabel). The review confirmed zero findings; its minors were fixed (picker retention, anchor-failure alert, double reload on window open). Ground truth: 186 tests green and ByteLife.app 0.5.0 running. The share path needs the founder's manual click to confirm end to end. Next step: founder verdict on the Back Office and the share button, then the next layer (weekly Statement, holdings schema groundwork, or the rename).

## 2026-07-07 — Iteration 7: instant panel, LIVE control, Messages-proof share, granularity, and the brew tap, version 0.6.0

The plan for this iteration came from founder feedback on 0.5.0: a Homebrew tap for installation, no slide-in or delayed LIVE state when the panel opens, sharing that actually reaches Messages, period granularity beyond single days, and a LIVE toggle with strictly periodic background refreshing.

Progress: a three-stage workflow built the app changes. The panel now paints its first frame unanimated and opens warm (the thirty-second background tick carries the snapshot and EMA state, and reopen only cold-starts after a gap over forty-five seconds), so LIVE is correct immediately; the LIVE chip became a persisted toggle that starts or stops the fast two-second cadence while open. Sharing now routes through a dedicated Receipt window, because compose targets like Messages attach their session to the host window and the menubar panel dies on activation (Photos worked because it has no UI); the window auto-presents the share picker once front and key. The Back Office gained Day, Week, and Month granularity: ISO Monday-start weeks and calendar months grouped by a tested core model, aggregate stories with per-day bars, and posted-coverage chips that jump to the day. The adversarial review then proved a real leak in the peak gate — a long-gap average carried in the display EMA could be promoted to a peak by the next short tick — and peaks now come from a separate peak-only EMA that integrates only short-gap measurements and resets when the chain breaks, with an end-to-end regression test; two review minors (chronological aggregate bars across month boundaries, granularity-switch carryover) were also fixed. The tap shipped: scripts/release.sh builds, zips, publishes a GitHub release on the public homebrew-tap repo (byteslife stays private), and writes Casks/bytelife.rb in the tap's house style; verified by tapping, parsing, and fetching with checksum. Ground truth: 206 tests green, ByteLife.app 0.6.0 running, and brew install --cask bytelife live. The founder still needs to click Share to Messages once to confirm the stable-window fix end to end. Next step: founder verdict on the share click and granularity, then the next layer (weekly Statement receipts, holdings schema groundwork, or the rename).
