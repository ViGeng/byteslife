# Feasibility research: measuring the five metric families on macOS

> Raw research output from the 2026-07-04 concept workflow (14 agents). Preserved verbatim for implementation reference; not edited for style.

## AI token usage tracking (Claude Code, ChatGPT/Codex, Cursor, GitHub Copilot, local LLMs via Ollama) for the ByteLife macOS menubar app

**Recommended approach.** Default to local JSONL/log parsing for each tool the user has installed, no proxy and no org-level API keys required: watch ~/.claude/projects/**/*.jsonl for Claude Code (summing message.usage.input_tokens/output_tokens/cache_creation_input_tokens/cache_read_input_tokens per assistant message, deduplicating by message id or by (sessionId, uuid) since duplicate lines were observed empirically), watch ~/.codex/sessions/**/rollout-*.jsonl for Codex CLI (diffing cumulative token_count events), and poll Ollama's /api/generate and /api/chat responses via a lightweight local reverse-proxy wrapper (or, more simply, note that Ollama itself has no log, so this only works if ByteLife intercepts the loopback traffic to 127.0.0.1:11434, which is a benign, low-trust proxy case since it's plaintext localhost traffic the user's own machine originates, not a real MITM of third-party HTTPS). For a non-sandboxed direct-distribution build, these directories can be watched continuously via FSEvents/DispatchSource file-monitoring with zero prompts; for a Mac-App-Store build, prompt the user once per tool to pick the relevant folder (~/.claude, ~/.codex, etc.) via NSOpenPanel and persist a security-scoped bookmark. Treat Cursor and GitHub Copilot as best-effort/unsupported for true token counts (Copilot doesn't expose tokens at all, only request counts; Cursor's local data is unreliable) -- surface whatever coarse numbers their official dashboards/APIs give (request counts, premium-request quota) rather than pretending to have token-level precision. Offer the Anthropic/OpenAI Admin APIs only as an optional power-user integration for people who happen to run a one-person organization and generate themselves an Admin key, clearly labeled as unavailable to most personal accounts.

**Fallback.** When no local transcript file exists for a tool (e.g. Cursor, or Copilot without OTEL enabled, or ChatGPT web usage with no API), fall back to manual/self-reported entry: let the user periodically paste in a number from the vendor's own web usage page, or, for Claude Code power users, let them opt into hook-script based logging (Claude Code's documented Stop/PostToolUse hooks) that writes usage into a file ByteLife also watches. As a last-resort, advanced-only, non-App-Store mode, offer a local HTTPS forward proxy the user can point supported CLIs at (via HTTPS_PROXY env var or tool-specific proxy config) after manually trusting a generated root CA in Keychain Access -- clearly gated behind an explicit warning screen, disabled by default, and never proposed for the Mac App Store build given the trust and certificate-pinning issues.

**Permissions.**
- No TCC (Transparency/Consent/Control) prompt exists for reading arbitrary dotfile directories such as ~/.claude/projects, ~/.codex/sessions, or ~/.cursor -- these are not among the TCC-protected locations (Documents, Desktop, Downloads, Photos, etc.). A non-sandboxed (Developer-ID-signed, direct-distribution) app can read them with plain POSIX permissions and no prompt at all.
- A sandboxed (Mac App Store) app cannot read those same dotfile directories in the background without the user first granting access via an NSOpenPanel (standard Open dialog) pointed at the folder, because App Sandbox only auto-grants access to the app's own container plus a few standard folders (Downloads, Pictures, etc.) via entitlements like com.apple.security.files.downloads.read-write -- arbitrary paths like ~/.claude require the user-selected-file entitlement (com.apple.security.files.user-selected.read-write) triggered by an open panel.
- After that one-time user grant, a sandboxed app can persist the access across launches using a security-scoped bookmark (com.apple.security.files.bookmarks.app-scope), calling startAccessingSecurityScopedResource on each access -- so continuous background monitoring is possible for a sandboxed app, but only after an explicit, one-time user action picking that folder, not silently.
- com.apple.security.network.client entitlement is required for any sandboxed app to make outbound HTTPS calls (to Anthropic's Admin API, OpenAI's Usage API, or a local Ollama server) -- this is a standard, uncontroversial entitlement freely available to Mac App Store apps.
- Admin API key (sk-ant-admin01-...) for Anthropic's Usage/Cost and Claude Code Analytics APIs -- explicitly gated behind having set up a Claude Console organization; not available to individual/personal accounts at all, regardless of sandboxing.
- OpenAI Usage API is scoped to a 'Default Organization' and API keys; individual ChatGPT subscription usage is not exposed by any API found.
- Running a local MITM/forward proxy requires the user to manually install and set 'Always Trust' on a self-signed root CA certificate in the login or System keychain -- this is a manual Keychain Access step Apple does not let an app perform silently for security reasons, and it is a meaningfully higher trust ask than any other option here.

**App Store compatibility.** Partial. The App Sandbox itself is not the real obstacle for the local-file-parsing approach (Claude Code/Codex JSONL logs): a sandboxed app can read them, but only after the user grants folder access once via a standard Open panel, which then persists via a security-scoped bookmark -- this is a one-time friction point, not a hard block, and is a normal, App-Store-legal pattern (similar to how sandboxed text editors get access to arbitrary project folders). Calling the Anthropic/OpenAI usage APIs or a local Ollama server needs only the standard, always-approved com.apple.security.network.client entitlement, so no App Store issue there. The one approach that is flatly incompatible with App Store distribution is the local MITM/HTTPS-proxy fallback: silently installing a trusted root CA is not something Apple permits an app to do without heavy, explicit user action in Keychain Access outside the app's control, and Apple review would likely flag covert traffic interception; that mode, if built at all, should be confined to a non-MAS (Developer ID, direct-download) build or an explicit advanced opt-in with the user performing the Keychain trust step themselves.

**Energy cost.** Continuously tailing a handful of JSONL files via kqueue/FSEvents (macOS's native file-system event notification, not polling) is very cheap: the OS wakes the process only on file writes, and Claude Code/Codex sessions write infrequently (per assistant turn, not per token), so realistic CPU usage is negligible (well under 1% average, brief wake-ups only). Parsing appended JSONL lines incrementally (seeking to the last read offset rather than re-reading whole files) keeps this cheap even for large, long-running session logs. The Ollama-proxy approach is more costly in engineering complexity than CPU: it must proxy every local inference request, which added negligible latency for local traffic already, but it necessarily makes ByteLife a man-in-the-middle for the user's own local LLM tooling. The MITM-everything last-resort proxy is the most expensive of the options, both because all HTTPS traffic sees added latency for interception and re-encryption, and because it requires the process to be always-on for any of the user's AI tools to keep functioning, which for a menubar app that might crash or be killed represents a reliability risk to unrelated network traffic, not merely an energy cost. Polling the Anthropic/OpenAI Admin APIs, where available, is trivially cheap (their own docs recommend once-per-minute polling at most).

**Candidate APIs.**
- `Claude Code local JSONL transcripts (~/.claude/projects/**/*.jsonl)` (file/log parsing): Confirmed empirically on this machine. Each project gets a directory under ~/.claude/projects (name derived from the working-directory path with slashes replaced by dashes), containing one JSONL file per session (named by session UUID). Records have type values including 'user', 'assistant', 'queue-operation', 'attachment'. Assistant records carry message.usage with input_tokens, output_tokens, cache_creation_input_tokens, cache_read_input_tokens, and a nested cache_creation breakdown (ephemeral_1h_input_tokens, ephemeral_5m_input_tokens). Top-level fields include timestamp, sessionId, cwd, version, uuid, parentUuid. Caveat found in this session's own data: consecutive assistant lines can carry byte-for-byte identical usage blocks and timestamps (e.g. tool-call retries or same-turn duplicates), so a naive per-line sum overcounts; real parsers must deduplicate, and ccusage is documented to do so by message/request identity though its exact dedup key was not independently confirmed from source (GitHub raw README/source fetch 404'd). This is the same JSONL source ccusage, tokscale, and other community tools read. Some environments alternatively use ~/.config/claude/projects per one secondary source; this was not verified against Anthropic's own documentation and should be treated as unconfirmed.
- `ccusage (open-source CLI, npm package + GitHub ryoppippi/ccusage)` (CLI tool parsing): Confirmed via npm, GitHub, and ccusage.com. Parses local JSONL files (no network calls to Anthropic needed) to produce daily/weekly/monthly/session/5-hour-window reports with cost estimates from LiteLLM pricing data. Beyond Claude Code, current versions also parse local logs for Codex CLI, OpenCode, Amp, Droid, Codebuff, Gemini CLI, GitHub Copilot CLI, and others, making it a useful reference implementation for a multi-tool parser. Could be shelled out to, vendored, or reimplemented in Swift; reimplementing in Swift avoids a Node.js runtime dependency, which matters for a lightweight menubar app.
- `Anthropic Usage & Cost Admin API (/v1/organizations/usage_report/messages, /v1/organizations/cost_report)` (third-party service): Confirmed via official Anthropic docs (platform.claude.com, migrated from docs.anthropic.com). Returns per-day/hour/minute token buckets grouped by model, workspace, API key, service tier. EXPLICITLY STATED: 'The Admin API is unavailable for individual accounts' -- requires setting up an organization in Claude Console and an Admin API key (sk-ant-admin01-...), a different credential from a normal API key. Data freshness ~5 minutes; polling once/minute is the documented supported rate. This confirms the lead: personal/individual Anthropic accounts cannot use this endpoint.
- `Claude Code Analytics Admin API (/v1/organizations/usage_report/claude_code)` (third-party service): Confirmed via official docs. Gives daily per-user token/cost/productivity breakdowns specifically for Claude Code, but carries the identical restriction: 'The Admin API is unavailable for individual accounts,' requires an organization and Admin API key. Not usable for a single-user consumer app unless the user happens to run their own one-person Anthropic organization and manually issues themselves an Admin key -- worth noting as a possible opt-in power-user path but not a default flow.
- `OpenAI Usage API (/v1/organization/usage/{completions,...}) and organization Usage dashboard` (third-party service): Confirmed via web search of OpenAI help docs. Endpoint is organization-scoped. Individual ChatGPT Plus/Pro subscription usage (chat.openai.com / chatgpt.com token consumption) is not exposed via this API at all -- it only covers API-key-based usage tracked under a Default Organization, and only for keys created after Dec 2023 or opted into tracking. There is no confirmed way to pull ChatGPT-web token counts programmatically; this matches the domain lead that OpenAI's usage/cost APIs have similar org/API-key gating to Anthropic's, and are largely unusable for a personal ChatGPT (non-API, non-org) subscriber.
- `Codex CLI local session logs (~/.codex/sessions/**/rollout-*.jsonl, also archived_sessions/)` (file/log parsing): Confirmed via GitHub issues/community sources referencing OpenAI's own codex repo and via ccusage's Codex-specific docs. Events of type token_count carry cumulative totals (total_tokens, input_tokens, cached_input_tokens, output_tokens, reasoning_output_tokens); per-turn deltas require subtracting successive cumulative snapshots. turn_context in the session identifies the active model. ccusage explicitly skips early-format sessions lacking model metadata. This is a solid, real local data source for Codex CLI specifically -- not for ChatGPT's own product surfaces.
- `Cursor local state (~/.cursor, ~/Library/Application Support/Cursor/User/globalStorage/state.vscdb)` (file/log parsing): Investigated but found to be poor quality as a source. Unlike Claude Code, Cursor has no single append-only per-session JSONL log; usage/timing signals are scattered across a large SQLite state database (state.vscdb) mixed with unrelated editor state, checkpoints, and file history, with coverage described by community sources as 'uneven.' The community tool tokscale does not parse Cursor's local files for token counts at all -- instead it caches Cursor's own authenticated web/API usage export (~/.config/tokscale/cursor-cache/usage*.csv), i.e. it falls back to Cursor's hosted usage dashboard/API rather than local parsing. No official, stable Cursor local-log token schema could be confirmed.
- `GitHub Copilot billing/usage settings and premium-request analytics API` (third-party service): Confirmed via GitHub docs and changelog. Copilot bills and reports in 'requests' (interactions), not raw token counts -- there is no per-token usage figure exposed to users at all, for individual or org accounts. Individual users can see premium-request counts at github.com/settings/billing or in-editor, and GitHub now has a premium-request analytics API, but it is aggregated, not per-token, and its per-user breakdown is limited even for enterprise orgs per a linked GitHub community issue. Separately, GitHub Copilot CLI (the newer standalone CLI) can emit OpenTelemetry JSONL logs (~/.copilot/otel/*.jsonl per tokscale) if OTEL export is explicitly enabled, which is the only path to token-level Copilot data found, and it requires the user to turn on telemetry export themselves.
- `Ollama HTTP API (/api/generate, /api/chat response fields prompt_eval_count, eval_count; /api/ps for loaded-model status)` (public framework): Confirmed via official Ollama docs (docs.ollama.com/api/generate) fetched directly. prompt_eval_count = input tokens, eval_count = output tokens, both returned per-request in the JSON response body, alongside duration fields in nanoseconds for computing tokens/sec. Ollama has NO built-in history, log, or aggregation endpoint for cumulative usage over time -- confirmed explicitly by the fetched docs page. A known bug (GitHub ollama/ollama#3427) causes prompt_eval_count to be absent or wrong when the prompt exceeds num_ctx. To get cumulative daily token counts from Ollama, ByteLife would have to sit in front of every request as a local reverse proxy on 127.0.0.1:11434 (Ollama's default bind address) and sum the fields itself, since Ollama does not log this anywhere on disk by default.
- `Local HTTPS forward/MITM proxy for arbitrary AI tool traffic (e.g. mitmproxy-style local redirect)` (CLI tool parsing): Investigated as the last-resort catch-all. Confirmed via mitmproxy's own docs and multiple write-ups: macOS does not trust a locally generated CA by default, so the user must manually import and mark it 'Always Trust' in Keychain -- a significant, security-relevant ask for a menubar utility, and one that itself creates a supply-chain single point of failure (anything trusting that CA can decrypt all of the user's HTTPS traffic, not just AI traffic). Equally important: any client using certificate pinning (increasingly common, and something Anthropic/OpenAI or their SDKs could adopt) is immune to this technique entirely, so it cannot be considered a reliable general solution. This should be framed strictly as an opt-in, advanced/last-resort mode, never a default.
- `Manual/self-reported logging (user-entered token counts, or a Shortcuts/CLI hook the user wires up themselves)` (file/log parsing): Not an API at all; included because the domain brief names it as a last resort. No source needed beyond noting Claude Code supports hook scripts (documented Anthropic feature) that fire on events like Stop or PostToolUse and could, per-user-configuration, append usage data somewhere ByteLife reads -- this is opt-in wiring by the user, not something ByteLife can do unilaterally.

**Caveats.**
- Consecutive assistant lines in a single Claude Code JSONL session can carry byte-identical usage blocks and timestamps (observed directly in this session's own transcript file), so naive per-line summation will overcount actual token consumption; a real implementation needs a dedup key, and the exact dedup strategy used by ccusage could not be independently confirmed because its GitHub source file could not be fetched (404) and its docs page did not specify the mechanism.
- Whether some environments use ~/.config/claude/projects instead of ~/.claude/projects as an alternate path (mentioned by one secondary source, ccusage's own guide page fetch) was not corroborated against Anthropic's official documentation and should be treated as unconfirmed; the canonical, empirically-verified path on this machine is ~/.claude/projects.
- Ollama's prompt_eval_count field is documented to be missing or wrong when the prompt exceeds the model's configured num_ctx context window, so token counts from local models can silently undercount for long-context sessions.
- Cursor and GitHub Copilot do not have a reliable, officially documented local file format for token-level usage; any ByteLife support for them will be best-effort, coarser (request counts, not tokens for Copilot), or dependent on scraping vendor dashboards/APIs that were designed for humans, not machine polling.
- The Anthropic and OpenAI Admin/organization usage APIs are confirmed unavailable to individual, non-organization accounts, so the large majority of ByteLife's target users (individual Claude Pro/Max, ChatGPT Plus/Pro, or pay-as-you-go API users without an org) cannot use these APIs at all; local file parsing is not an inferior alternative for these two vendors, it is the only viable one for most users.
- One WebFetch result describing Apple's App Sandbox default file-access rules could not retrieve live page text (the fetch tool reported the page returned only a title) and fell back to general knowledge; this was cross-checked against a separate, successful WebSearch result confirming the same entitlement names and behavior, but it is weaker sourcing than the other findings in this report and is flagged accordingly.
- A live web-search classifier ('claude-opus-4-8[1m] temporarily unavailable') intermittently blocked WebSearch/WebFetch calls during this research session; all facts reported here reflect calls that did succeed, but some adjacent avenues (e.g., deeper Codex/ccusage source-code inspection) were left unexplored once a 404 was hit rather than retried indefinitely.

## Network traffic volume tracking (global interface byte counters and per-app attribution) for a macOS 14+ menubar app

**Recommended approach.** For the core dashboard metric (total bytes sent/received across all interfaces), use sysctl(NET_RT_IFLIST2) reading struct if_msghdr2 / if_data64 for 64-bit counters, polled on a timer (e.g., every 1-5 seconds) and persisted as deltas to build a running total that survives the 4 GiB wrap of any single sampling window. This avoids getifaddrs' 32-bit overflow problem entirely and is a public, sandboxable POSIX-level API requiring no entitlements or user consent. Sum across all non-loopback interfaces (or let the user pick Wi-Fi/Ethernet only) to get the headline up/down byte counts. Do not attempt per-app network attribution in v1: every available option (nettop-parsing, NetworkExtension content filter, private frameworks) carries real cost — undocumented CLI dependency, an Apple-approved entitlement plus mandatory user-facing System Extension consent, or App Store rejection risk, respectively — that is disproportionate to a 'nice-to-have' breakdown feature in a dashboard whose core promise is aggregate totals.

**Fallback.** If per-app attribution becomes a hard product requirement later, the least-bad path is spawning nettop as a background subprocess with `-P -L 0 -x -J bytes_in,bytes_out` (or similar column selection) and parsing its continuous CSV stream, since this requires no entitlement, no user consent dialog, and no private framework linkage from the app's own binary, at the cost of depending on an undocumented, Apple-owned CLI contract that could change between OS versions without notice, and needing careful handling of nettop's own cumulative-since-launch counter semantics (the app would need to keep its own nettop child process running continuously to get a stable baseline, or diff each restart). If Apple-sanctioned exact accounting is required instead of a best-effort estimate, the NetworkExtension content-filter path (NEFilterDataProvider/NEFilterPacketProvider as a System Extension) is the only fully public, App-Store-legal route, but it should be scoped as a distinct, opt-in feature given its Apple approval step and the explicit System Settings consent screen the user must click through.

**Permissions.**
- No entitlement or TCC prompt found for getifaddrs() or sysctl(NET_RT_IFLIST2) — these are standard POSIX/BSD calls; global interface byte totals should work in a default App Sandbox without special entitlements, though I could not find an Apple source stating this explicitly for the sandbox profile and flag this as needing direct empirical confirmation in a sandboxed test build.
- nettop requires no elevated privilege and needs no entitlement to invoke as a subprocess, confirmed by direct local testing as a standard (non-root, non-admin-elevated) user.
- NetworkExtension content filter on macOS requires the com.apple.developer.networking.networkextension entitlement with array value 'content-filter-provider-systemextension', plus com.apple.developer.system-extension.install. Multiple developer forum threads describe needing to email networkextension@apple.com to request Apple's approval for this entitlement class before Xcode/provisioning will issue it; approval is not automatic.
- Installing the resulting System Extension on macOS requires explicit user approval in System Settings > Privacy & Security (a system consent dialog naming the extension and its developer), which the user must approve manually outside the app's own UI unless deployed via MDM (which can suppress the prompt only in managed/enterprise contexts, not applicable to a consumer app).
- Private frameworks such as NetworkStatistics.framework require no public entitlement because they are not a sanctioned public API at all; there is no legitimate permission path for a third-party app to declare it wants to use them.

**App Store compatibility.** Partial. Global interface-level byte counters (getifaddrs/sysctl) are plain POSIX calls with no known sandbox restriction and are almost certainly Mac App Store safe, though I could not find an explicit Apple sandbox-profile statement confirming this and recommend a direct sandboxed-build smoke test. Per-app attribution is the part that is NOT cleanly App Store compatible: nettop-parsing depends on an undocumented, unversioned CLI tool whose format Apple could change and whose long-term MAS acceptability is unverified (I found no confirmed example of a shipping MAS app that parses nettop); the NetworkExtension content-filter path is a public, sanctioned API but requires a discretionary Apple-approved entitlement, a System Extension (which historically has faced approval friction for non-security/parental-control use cases per forum reports), and an explicit user consent dialog outside the app; and NetworkStatistics.framework is a private framework that Apple's static/dynamic App Store scanning is documented to detect and reject (the ITMS-90338-style private-API-usage rejection pattern is well established for any private framework, per Apple's public Xcode/App Store Connect error codes and multiple cross-checked developer reports).

**Energy cost.** Polling sysctl(NET_RT_IFLIST2) or getifaddrs on a multi-second timer (e.g., every 1-5 seconds) is a lightweight, non-blocking syscall with negligible CPU and energy cost, comparable to what iStat-Menus-style menubar monitors already do continuously; I did not find a published benchmark number, but the operation is a single small kernel-table read with no network I/O of its own, so the cost is dominated by how often the app wakes up and redraws its UI rather than by the syscall itself. Using a longer interval (matching typical menubar-monitor UX of 1-2 second refresh) and coalescing UI updates should keep this comfortably within menubar-app energy-impact norms shown in Activity Monitor. Continuously spawning and parsing nettop as a long-lived subprocess is more expensive than a syscall (a persistent child process plus continuous text parsing) and would show up as its own line in Activity Monitor's Energy tab; exact overhead was not benchmarked in any source found and should be measured directly if that fallback path is implemented.

**Candidate APIs.**
- `getifaddrs() + struct if_data (32-bit ifi_ibytes/ifi_obytes)` (public framework): POSIX/BSD API in Darwin libc, directly callable from Swift. Gives per-interface cumulative bytes in/out since boot, but the counters are u_int32_t and wrap at 4 GiB. Apple engineers on the Developer Forums (thread 712340) describe this legacy interface as something macOS inherited that 'doesn't really make much sense in the modern world' and suggest filing an enhancement request rather than relying on it. Some reports describe doubled ibytes values on macOS 11/12 versus macOS 10. Does not expose the 64-bit if_data64 fields.
- `sysctl(NET_RT_IFLIST2) returning struct if_msghdr2 with embedded struct if_data64` (public framework): Confirmed working approach for 64-bit interface byte counters (ifi_ibytes/ifi_obytes as u_int64_t), avoiding the 32-bit overflow of getifaddrs. Documented in a detailed technical writeup (milen.me) with an accompanying open-source Swift/C reference implementation (github.com/milend/macos-network-metrics). Caveat found in that source: as of macOS Ventura 13.2.1 there is a kernel bug (tracked as rdar://106029568) that truncates/wraps traffic counters at 4 GiB for third-party (non-Apple-signed) processes, and the kernel also batches/quantizes traffic values to 1 KiB granularity for third-party programs specifically to prevent fingerprinting. An alternative sysctl leaf, IFMIB_IFDATA, is reported to avoid both the batching and the truncation bug, but this is a single-source claim I could not cross-verify against a second independent source.
- `nettop CLI (per-process bytes_in/bytes_out, CSV output via -L/-J flags)` (CLI tool parsing): Verified directly on this machine (macOS, current user, no sudo): `nettop -P -L 1 -x -J bytes_in,bytes_out` produces CSV lines like `mDNSResponder.497,126127339,105669408,` giving per-process cumulative bytes since nettop started, keyed by 'processname.pid'. Confirmed nettop requires no root/sudo privilege (unlike lsof). This is a legitimate, scriptable per-app attribution path but requires spawning and continuously parsing a subprocess, and Apple could change the CLI's column names/format without notice since it is undocumented as a stable API. nettop internally uses the private NetworkStatistics.framework (per an *OS Internals reference cited in search results), which nettop itself is permitted to use as an Apple-signed system binary; a third-party app parsing nettop's stdout does not link that private framework itself, so this path avoids the App Store private-API linkage problem while still getting the same data indirectly.
- `NetworkStatistics.framework (nstat* / NSTAT symbols)` (private API): Private framework used internally by Activity Monitor and nettop to obtain per-process network statistics. Referenced in web search results citing the '*OS Internals, Volume I' book as having a chapter documenting the private API, and an open-source clone called 'netbottom' reportedly demonstrates using these private APIs directly. I could not fetch primary documentation of the exact symbol names (e.g., nstat_manager) or a maintained public header; treat exact API surface as unconfirmed pending direct source inspection of a project like netbottom.
- `NetworkExtension (NEFilterDataProvider / NEFilterPacketProvider) content filter` (public framework): Apple's supported way to observe per-flow network traffic (TCP/UDP flows, and for NEFilterPacketProvider, Layer 2 packets) system-wide, including per-flow app attribution via properties on NEFilterFlow. Confirmed via Apple's own documentation pages (titles retrieved; full body text was not renderable through the fetch tool used, so field-level detail like exact property names is not independently confirmed here and should be re-checked directly in Xcode's API docs). Confirmed via a real implementation report (Apple Developer Forums thread 769199) that macOS requires two entitlements together: com.apple.developer.networking.networkextension with the array value 'content-filter-provider-systemextension' (note this differs from the iOS-only value 'content-filter-provider'), plus com.apple.developer.system-extension.install. This is a System Extension architecture (not an App Extension) on macOS, distinct from iOS's app-extension-based filter. Filter Data Provider extensions run in a highly restrictive sandbox that itself cannot access the network or write to disk, per Apple's documentation summary in search results.

**Caveats.**
- getifaddrs' ifi_ibytes/ifi_obytes are 32-bit and wrap at 4 GiB; on any Mac with meaningful daily traffic this will wrap in hours, so getifaddrs alone is unsuitable as the sole data source for a long-running total.
- sysctl(NET_RT_IFLIST2)'s 64-bit if_data64 counters reportedly still suffer a macOS Ventura 13.2.1 kernel bug (rdar://106029568) that truncates and wraps at 4 GiB specifically for third-party (non-Apple-signed) processes; I found this claim in one detailed technical source (milen.me) and it was not independently cross-verified against a second source, so treat the exact OS-version scope of the bug as unconfirmed.
- The kernel reportedly quantizes/batches traffic values to 1 KiB granularity for third-party (non-Apple-signed) programs specifically to reduce fingerprinting risk, meaning even correct 64-bit reads will have some inherent imprecision versus what an Apple-signed tool like Activity Monitor sees; this is a single-source claim (same milen.me article) not independently verified.
- Software-reported interface byte counters (whichever API is used) count bytes at the network-interface/socket layer, not necessarily true over-the-wire physical bytes; this is a general known limitation of all software counters versus router/modem-side counters, not specific to any one API here.
- The nettop CLI output format (column names, delimiter behavior, per-process cumulative-since-launch semantics) is undocumented as a stable contract; the man page confirms the flags used (-P, -L, -J, -x) but Apple could change formatting in a future macOS release.
- I could not directly confirm NetworkStatistics.framework's exact public symbol names or a maintained header; the private-API claim rests on secondary sources (an *OS Internals book reference and a forum mention of a 'netbottom' clone project) rather than a primary Apple source, which is expected since it is undocumented by definition, but means the specific API surface described here should be verified against the actual framework binary or the netbottom source before implementation.

## Disk read/write volume tracking (system-wide and per-process bytes read/written) on macOS 14+

**Recommended approach.** Poll IOKit's IOBlockStorageDriver Statistics dictionary for system-wide (per-physical-disk) cumulative bytes read/written: enumerate physical disks with IOServiceMatching("IOMedia") + IOServiceGetMatchingServices(kIOMainPortDefault, ...), or resolve a specific mounted volume's BSD name via DiskArbitration (DADiskCreateFromVolumePath/DADiskGetBSDName) and then IOServiceGetMatchingService(kIOMainPortDefault, IOBSDNameMatching(...)); walk to the parent registry entry with IORegistryEntryGetParentEntry until IOObjectConformsTo(parent, "IOBlockStorageDriver") is true (this typically requires walking past IOMedia/CoreStorage/APFS container layers to the physical driver, as exelban/stats does with a partition-count-derived 'level' parameter); read the "Statistics" dictionary via IORegistryEntryCreateCFProperties and pull the "Bytes (Read)"/"Bytes (Write)" OSNumber values; diff consecutive samples (e.g., every 1-2 seconds) to get instantaneous rates and accumulate deltas for a running total. Layer in per-process attribution by calling proc_pid_rusage(pid, RUSAGE_INFO_CURRENT, &buf) and reading ri_diskio_bytesread/ri_diskio_byteswritten for each PID obtained via proc_listallpids (a public libproc call), not by shelling out to /bin/ps, to stay Sandbox-clean; diff samples per PID the same way. Ship unsandboxed first (direct distribution/notarized, matching the founder's own tool's likely initial distribution) to avoid the sandbox-compatibility unknown entirely, and separately validate the sandboxed path (test iokit-get-properties access to IOBlockStorageDriver empirically, or apply for a temporary-exception entitlement) before attempting an App Store submission.

**Fallback.** If IOKit Statistics access is ever denied under the sandbox (deny(1) iokit-get-properties in the sandbox violation log), fall back to system-wide-only tracking via proc_pid_rusage summed across all same-UID PIDs (proc_listallpids) as a proxy for total disk I/O, accepting that this misses kernel-level I/O not attributable to a user process (e.g., some filesystem journal/metadata traffic, Spotlight indexing edge cases, or other users' processes) and therefore undercounts true physical-disk bytes versus the IOKit path. A secondary, coarser fallback is periodically shelling out to `iostat -d` and parsing its text output, which reads the same IOBlockStorageDriver counters Apple's own tool exposes, at the cost of process-spawn overhead, text-parsing fragility across macOS versions, and definite App Sandbox incompatibility (process launching is restricted).

**Permissions.**
- No TCC (Transparency, Consent, and Control) prompt is triggered for either the system-wide IOKit Statistics read path or for proc_pid_rusage on same-UID processes; these are not among the privacy-protected resource categories (unlike Full Disk Access, which governs file content access, not I/O byte counters).
- No entitlement is required for the system-wide IOKit registry read path in a non-sandboxed app: Apple's iostat and exelban/stats both call IOServiceGetMatchingServices/IORegistryEntryCreateCFProperties as an ordinary unprivileged user process with no AuthorizationRef, no privileged helper, and no elevated persona.
- proc_pid_rusage requires no entitlement or privilege for processes owned by the same UID as the caller (confirmed via the CHECK_SAME_USER branch in xnu's proc_security_policy); querying a process owned by a different UID (e.g., root-owned daemons, another logged-in user) returns EPERM unless the caller itself has root's implicit PRIV_GLOBAL_PROC_INFO privilege.
- Under the App Sandbox specifically, IOKit registry reads are gated by the 'iokit-get-properties' sandbox operation. Multiple public sandbox-violation reports (Flutter, Mozilla, WebKit bug trackers) show iokit-get-properties denials occurring for various IOKit classes when sandboxed without the matching allow rule; the default sandbox profile allows a limited default set of IOKit classes and it is not independently confirmed from primary sources whether IOBlockStorageDriver/IOMedia are in that always-allowed default set or require a com.apple.security.temporary-exception.iokit-user-client-class entitlement (which App Review discourages and can query-reject) — flagged as a caveat needing an empirical sandboxed-build test.
- Real-world evidence leans favorable: both iStat Menus 7 (App Store id6499559693) and MenuBar Stats (App Store id714196447) ship on the sandboxed Mac App Store while advertising disk activity monitoring in their listings, implying at least the system-wide IOKit statistics path is achievable inside the sandbox, though neither app's entitlements plist was directly inspected to confirm zero use of temporary-exception entitlements.
- exelban/stats itself ships fully unsandboxed: its Stats.entitlements file contains no com.apple.security.app-sandbox key at all, only com.apple.security.cs.disable-library-validation and an application-groups entitlement for its widget extension. It is distributed via direct download/Homebrew/GitHub releases, not the Mac App Store, so it is not a data point for sandbox compatibility, only for the non-sandboxed baseline.

**App Store compatibility.** Partial — system-wide disk read/write totals via IOBlockStorageDriver Statistics are plausibly sandbox-compatible (iStat Menus 7 and MenuBar Stats both ship this on the App Store today), but this could not be independently confirmed from primary Apple sandbox-profile sources naming IOBlockStorageDriver/IOMedia as default-allowed iokit-get-properties targets, so an empirical sandboxed-build test is needed; per-process diskio via proc_pid_rusage should work sandboxed for same-UID processes with no extra entitlement since the kernel check is UID-based, not sandbox-based, but enumerating all PIDs by shelling out to /bin/ps (as exelban/stats does) is itself an App Sandbox violation (process launching of arbitrary system binaries is restricted) and would need to be replaced with proc_listallpids/proc_listpids.

**Energy cost.** Both the IOKit Statistics read and proc_pid_rusage are cheap, synchronous, non-blocking mach-message/syscall round-trips (no disk I/O of their own, no busy-polling of hardware); the practical energy cost is dominated by polling frequency and process-enumeration breadth rather than the calls themselves. Polling once every 1-2 seconds for a handful of physical disks (as exelban/stats does, setInterval(1)) is negligible. The per-process path is more expensive because it must enumerate and query every running process each cycle (proc_listallpids + one proc_pid_rusage call per PID, commonly 200-500+ processes on a typical Mac); doing this every 1 second is measurably more CPU-active than the system-wide path alone, and general power-management guidance (Apple's Power Programming Guide, cited in search results) indicates that sub-100ms polling intervals keep the CPU in higher P-states, so a multi-second interval (2-5s) with adaptive backoff when idle is preferable to 1-second polling for a background-resident menubar app aiming for low energy impact. Shelling out to /bin/ps or iostat every cycle (as exelban/stats does for its process list) adds real process-spawn overhead (fork/exec cost, pipe I/O) that a proc_listallpids-based implementation would avoid.

**Candidate APIs.**
- `IOKit IOBlockStorageDriver "Statistics" property dictionary (kIOBlockStorageDriverStatisticsKey = "Statistics", with sub-keys kIOBlockStorageDriverStatisticsBytesReadKey = "Bytes (Read)" and kIOBlockStorageDriverStatisticsBytesWrittenKey = "Bytes (Write)", plus Operations/Retries/Errors/Latency/Total-Time variants for read and write)` (public framework): Declared in IOKit/storage/IOBlockStorageDriver.h (part of the IOStorageFamily kext headers, mirrored in apple-oss-distributions/IOStorageFamily on GitHub). Apple Developer Documentation has stub pages for kIOBlockStorageDriverStatisticsKey and 'IOBlockStorageDriver.h User-Space > Defines', confirming this is documented (if thinly) public API, not private/undocumented. Access pattern verified directly in Apple's own open-source iostat(8) (system_cmds/iostat/iostat.c): match IOMedia services with IOServiceMatching("IOMedia") via IOServiceGetMatchingServices(kIOMainPortDefault, ...), walk to the parent with IORegistryEntryGetParentEntry, confirm it IOObjectConformsTo("IOBlockStorageDriver"), then read the Statistics dictionary with IORegistryEntryCreateCFProperties. This is a cumulative, monotonically increasing (until reboot) counter per physical block-storage driver instance (i.e., per physical disk/SSD, not per volume/partition) — you diff two samples to get a rate. No sudo, AuthorizationRef, or privileged helper appears anywhere in Apple's iostat source. exelban/stats (Modules/Disk/readers.swift) uses the identical technique: IOServiceGetMatchingService + IOBSDNameMatching, then getIOProperties on a parent node found by walking up 'level' times via IORegistryEntryGetParentEntry, reading props['Statistics']['Bytes (Read)'/'Bytes (Write)'] as NSDictionary/Int64. Use kIOMainPortDefault, not the deprecated kIOMasterPortDefault (deprecated since macOS 12), for a macOS 14+ target.
- `libproc proc_pid_rusage(pid, RUSAGE_INFO_CURRENT, &rusage_info_t) reading ri_diskio_bytesread / ri_diskio_byteswritten` (public framework): Declared in libproc.h (bsd/kern/proc_info via libsyscall wrappers), marked __OSX_AVAILABLE_STARTING(__MAC_10_9, __IPHONE_7_0) — a stable public libSystem/libproc symbol, though it is a low-level BSD API without its own polished Apple Developer Documentation page (documented mainly via the header and man-adjacent comments). Verified directly in Apple's open-source xnu kernel source (bsd/kern/proc_info.c): the kernel implementation calls proc_security_policy(p, PROC_INFO_CALL_PIDRUSAGE, flavor, CHECK_SAME_USER), which permits the call unconditionally when the caller's UID matches the target process's UID, and otherwise requires the PRIV_GLOBAL_PROC_INFO privilege (root has this by default) or returns EPERM. Practical implication: an unprivileged app can read diskio_bytesread/byteswritten for any process owned by the same logged-in user without root, but not for other users' processes or root-owned daemons unless it is itself root. exelban/stats (Modules/Disk/readers.swift, ProcessReader class) uses exactly this call, enumerating PIDs by shelling out to /bin/ps -Aceo pid,args -r (a public CLI, not a private API) rather than proc_listpids, then calling proc_pid_rusage per PID and diffing samples over time to compute per-process read/write rates.
- `DiskArbitration framework (DASessionCreate, DADiskCreateFromVolumePath, DADiskGetBSDName, DADiskCopyDescription)` (public framework): Used by exelban/stats only to enumerate mounted volumes and resolve a volume's BSD name/media UUID/removability/filesystem type, not to read byte counters itself. It is the standard public bridge from a mounted-volume URL to the BSD device name needed to then locate the underlying IOBlockStorageDriver in the IORegistry.
- `iostat(8) / "iostat -d" CLI output parsing` (CLI tool parsing): Apple's own iostat(8) reads via the identical IOBlockStorageDriver Statistics path documented above (confirmed by reading its open-source iostat.c), so shelling out to it would only duplicate functionality already reachable directly via IOKit with no extra privilege benefit. Not recommended as a primary approach since it adds process-spawn overhead and text-parsing fragility for no permission gain; useful only as a manual verification/debugging cross-check.

**Caveats.**
- Both counters are cumulative since driver/process instantiation (reset on reboot for the IOKit path, reset on process launch for proc_pid_rusage), not since app-install; the app must persist its own last-seen baseline per disk/PID and handle counter resets (reboot, or PID reuse after a process exits) to avoid reporting spurious huge deltas or negative diffs.
- The IOKit Statistics dictionary is keyed per physical IOBlockStorageDriver instance, which corresponds to a physical disk, not a logical volume or APFS container; on an APFS Mac with multiple volumes in one container (the default since macOS 10.13), all volumes in that container share one underlying driver's Statistics, so per-volume attribution is not possible via this API — only per-physical-device totals. exelban/stats works around this by resolving to the physical parent via level-based IORegistryEntryGetParentEntry walks, effectively collapsing all APFS volumes on one disk into one figure.
- External volumes (USB/Thunderbolt drives, SD cards, network-mounted DMGs) each have their own IOBlockStorageDriver in the registry and appear/disappear from the registry on mount/unmount, so the polling code must handle dynamic device arrival/departure (exelban/stats does this by diffing the active BSD-name list every poll cycle) rather than assuming a fixed disk set at launch.
- Network-mounted filesystems (SMB/AFP/NFS) and some virtual/synthetic volumes are not backed by an IOBlockStorageDriver at all, so this technique only covers local physical storage, not network shares; a separate accounting path (e.g., statfs-based or none) would be needed if network-share I/O should count toward the 'disk' metric.
- proc_pid_rusage's diskio counters are reported to reflect only I/O the kernel attributes to that specific process's VFS/read-write syscalls; I/O performed on the process's behalf by a different process (e.g., a helper XPC service, a shared cache write by the kernel itself, or coalesced writeback done later by the VM/UBC) may not be attributed to the originating app, so summing per-process rusage across all processes will not exactly equal the system-wide IOKit total — the two numbers are complementary, not reconcilable to the byte.
- The IOKit Statistics dictionary's precise key set and its exact registry depth (how many parent hops to reach IOBlockStorageDriver) have shifted historically across macOS/APFS versions per community reports referenced by exelban/stats' own code comments (citing Apple's bless/BLAPFSUtilities.c for the parent-walk logic), so the parent-walk level heuristic should be tested across current macOS 14/15 disk configurations (internal APFS Fusion/SSD, external USB, Thunderbolt RAID) rather than assumed stable.
- Apple Developer Documentation pages for kIOBlockStorageDriverStatisticsKey and the IOBlockStorageDriver_h_user-space header exist but render as near-empty stubs via automated fetch (JS-rendered SPA), so their full text (deprecation notices, exact availability macros) could not be directly quoted here; the header-level and xnu-source-level confirmation is solid, but a human should sanity-check the live Apple Developer page for any recent deprecation notices before shipping.
- Whether IOBlockStorageDriver/IOMedia registry reads are in the sandbox's default-allowed iokit-get-properties set (vs. requiring a temporary-exception entitlement) was not confirmed from a primary Apple sandbox-profile source; this is inferred only indirectly from two App Store apps (iStat Menus 7, MenuBar Stats) advertising disk-activity features while presumably being sandboxed, which is suggestive but not verified against their actual entitlements files.

## Screen-on / attentive time tracking (macOS 14+, Swift, menubar app)

**Recommended approach.** Build attentive-time tracking from first principles rather than relying on Apple's own Screen Time data store. Use CGEventSourceSecondsSinceLastEventType(.combinedSessionState, kCGAnyInputEventType) polled or checked on a low-frequency dispatch timer (or driven by a lightweight CGEventTap in listen-only mode, which is Input-Monitoring-gated but sandbox-compatible, if finer-grained event timestamps are wanted for the Input metric family anyway) to determine whether the user is currently idle past a threshold (for example 2-5 minutes, matching common screensaver defaults). Combine this with NSWorkspace notifications (willSleepNotification/didWakeNotification for system sleep, screensDidSleepNotification/screensDidWakeNotification for display sleep) and DistributedNotificationCenter's com.apple.screenIsLocked/com.apple.screenIsUnlocked for screen-lock state, to build a state machine of attentive vs. not-attentive intervals. Sum attentive intervals per day for the screen-time metric. This approach ships cleanly in a sandboxed Mac App Store build with no Full Disk Access and no gated entitlement.

**Fallback.** If Apple Silicon-specific display-sleep detection proves unreliable via IODisplayWrangler, fall back to treating NSWorkspace's screen-sleep/wake notifications (rather than raw IOKit registry polling) as the sole source of display power state, since these are documented AppKit notifications rather than an architecture-sensitive IOKit path. If the product later wants to cross-check or enrich its self-tracked attentive time against Apple's own Screen Time totals for validation purposes, that would require distributing outside the Mac App Store (Developer ID + notarization) and prompting the user for Full Disk Access to read knowledgeC.db, clearly disclosing why, as a strictly optional/advanced feature rather than a core dependency, since the schema is undocumented, private, and has changed across OS versions.

**Permissions.**
- No TCC prompt found for CGEventSourceSecondsSinceLastEventType itself (it returns only an elapsed-time scalar, not event content); contrast with CGEventTap and NSEvent.addGlobalMonitorForEvents, which read actual keystroke/mouse content and require Input Monitoring or Accessibility respectively -- relevant if the same app also implements the Input metric family.
- Full Disk Access (a TCC-gated, user-granted permission set manually in System Settings) is very likely required to read knowledgeC.db, based on general TCC protection of app-support database files since Catalina and on real developer/user reports of 'Operation not permitted' errors and FDA setup instructions for comparable tools; I could not fully reconcile this against one source claiming no FDA is needed.
- Full Disk Access does not bypass App Sandbox: per Apple Developer Forum guidance, a sandboxed app granted FDA can still only access items within its own static/dynamic sandbox scope, so FDA alone would not actually unlock knowledgeC.db for a sandboxed app regardless of the TCC question above.
- com.apple.developer.family-controls entitlement requires a manual Apple approval request (multi-week turnaround) and appears functionally inoperable on macOS regardless of approval, per developer reports of runtime sandbox-restriction failures.
- No entitlement/permission identified for NSWorkspace sleep/wake/session notifications or the com.apple.screenIsLocked/Unlocked distributed notifications; these fire for any process without special privilege.

**App Store compatibility.** Partial. The idle-detection core (CGEventSourceSecondsSinceLastEventType) plus NSWorkspace/DistributedNotificationCenter sleep-wake-lock-screensaver notifications appear fully compatible with a sandboxed Mac App Store app and require no special entitlement. However, reading Apple's own knowledgeC.db Screen Time database is not App Store viable: it needs Full Disk Access, which App Review treats as a red flag and which does not even bypass App Sandbox's own scoping restrictions, and DeviceActivity/FamilyControls are gated behind a manually-approved entitlement that multiple sources report does not work on macOS at all regardless of approval. A ByteLife build that only uses the idle/sleep/wake/lock primitives could ship sandboxed on the Mac App Store; a build that also wants Apple's own per-app Screen Time breakdown would need to go outside the sandbox (Developer ID distribution) and request Full Disk Access from the user directly, accepting the associated UX and trust cost.

**Energy cost.** Continuous idle-time monitoring via a low-frequency timer calling CGEventSourceSecondsSinceLastEventType (for example every 10-30 seconds) is lightweight: it is a simple kernel query, not a full event tap, and Apple's own energy-efficiency guidance is specifically that recurring timers -- not simple periodic system calls -- are the main energy risk because they prevent the CPU from reaching deep idle states. A better-practice implementation would use an event-driven design (dispatch source or dispatch timer with a coarse leeway, or triggering checks only around window-manager/space-change and sleep/wake notifications) rather than tight polling, per Apple's Energy Efficiency Guide for Mac Apps guidance that idle wakeups are penalized more than steady CPU use and that timers should be replaced with event-driven APIs wherever possible. Sleep/wake/lock notifications themselves are push-based (NSWorkspace/DistributedNotificationCenter) and add negligible continuous cost since the app is woken only on actual state transitions rather than polling for them.

**Candidate APIs.**
- `CGEventSourceSecondsSinceLastEventType(_:_:)` (public framework): Core Graphics / Quartz Event Services, available since Mac OS X 10.4. Call with CGEventSourceStateID.combinedSessionState (or .hidSystemState) and CGEventType(rawValue: ~0) (kCGAnyInputEventType) to get seconds since the last keyboard/mouse/trackpad event. This is the standard idle-time primitive used by ActivityWatch's aw-watcher-afk, Chromium's idle_mac.mm, and many auto-away/screensaver-style tools. It returns only a scalar elapsed-time float; it does not capture event content. No permission prompt or entitlement has been reported for this call in any source found (unlike CGEventTap or NSEvent global monitors, which read event content and do require Input Monitoring or Accessibility respectively). Treat 'confirmed unprompted' as reasonably but not 100% certain since Apple does not document its exact TCC classification.
- `ioreg -c IOHIDSystem (HIDIdleTime property)` (CLI tool parsing): Older/alternate idle-time source: shell out to `ioreg -c IOHIDSystem` and parse the HIDIdleTime value (nanoseconds since last HID event), or query the IOKit registry directly via IOServiceGetMatchingService/IORegistryEntryCreateCFProperty in-process instead of shelling out. - `NSWorkspace.didWakeNotification / willSleepNotification / screensDidSleepNotification / screensDidWakeNotification / sessionDidBecomeActiveNotification / sessionDidResignActiveNotification` (public framework): AppKit/NSWorkspace notification center. Used to detect system sleep/wake and, separately, display sleep/wake and fast-user-switching session activation. Because these are string-keyed distributed notifications rather than documented API, Apple could change or remove them without notice; there is no official Apple documentation confirming them, so treat as a de facto standard rather than a guaranteed contract.
- `IODisplayWrangler / IOPowerManagement (ioreg registry, CurrentPowerState)` (CLI tool parsing): Used on Intel Macs to determine whether the display itself is asleep (as distinct from system sleep) by inspecting `ioreg -n IODisplayWrangler | grep IOPowerManagement` for CurrentPowerState=4 (awake). My assessment is that FDA is required in current macOS versions for any app without it already granted, and any earlier no-FDA-needed reports likely reflect an older macOS version, a differently scoped read, or a misconfiguration on the tester's system. This is unverified with certainty either way from official Apple documentation, since Apple does not document knowledgeC.db as a supported API at all -- it is a private, undocumented database whose schema has changed across OS versions and could change or be locked down further without notice.
- `DeviceActivity / FamilyControls / ManagedSettings frameworks` (public framework): These are real, documented Apple frameworks (developer.apple.com/documentation/familycontrols, /devicactivity, /managedsettings) but are iOS/iPadOS-focused Screen Time APIs gated behind the com.apple.developer.family-controls entitlement, which Apple manually reviews and approves per app (multi-week wait times reported, 3.5-4.5+ weeks). **Caveats.**
- The claim that knowledgeC.db is readable without Full Disk Access could not be fully verified; sources conflict, and my best assessment (based on general TCC protection of app-support databases since Catalina, and on real 'Operation not permitted' error reports) is that FDA is required in current macOS versions, but this is not certain and should be tested empirically on macOS 14+ before relying on it.

- The com.apple.screenIsLocked/com.apple.screenIsUnlocked distributed notifications are not part of Apple's documented public API surface (no header declaration or doc page confirmed) and could be changed or removed without notice, unlike the formally documented NSWorkspace notifications.
- IODisplayWrangler-based display-sleep detection is confirmed unreliable on at least one Apple Silicon Mac (M2 Pro) in a real user report; this is a known architecture gap, not a hypothetical one, and needs an Apple Silicon-specific fallback (pmset -g log parsing, or preferably just relying on NSWorkspace's screen-sleep notifications instead of IOKit registry polling).
- CGEventSourceSecondsSinceLastEventType's lack of a permission requirement is inferred from its universal unprompted use across many real tools (ActivityWatch, Chromium) and forum discussion of the Accessibility/Input-Monitoring boundary applying to event-content APIs, not from an explicit Apple statement that this specific call is unprotected; I was not able to find a direct Apple statement confirming this either way.
- Screensaver activation itself is not perfectly synonymous with user attentiveness or its absence: a screensaver can be manually triggered while the user is still present (e.g., via a hot corner) and idle time resets on any HID event, so brief involuntary movements (a cat on the trackpad) could register as 'attentive' when the user is actually away; this is a known accuracy limitation of any idle-timer-based approach, not specific to macOS.

## Keystroke count and mouse travel distance tracking (macOS, listen-only, count-only, never recording which keys were pressed)

**Recommended approach.** Use CGEvent.tapCreate with place: .cgSessionEventTap (or .cgHIDEventTap), options: .listenOnly, and an eventsOfInterest mask covering keyDown/keyUp/flagsChanged (for keystroke counts) and mouseMoved/leftMouseDragged/rightMouseDragged/otherMouseDragged (for cumulative mouse travel distance, summed from event deltas). Gate creation on CGPreflightListenEventAccess()/CGRequestListenEventAccess() (or the IOHIDCheckAccess/IOHIDRequestAccess equivalents) so the app only ever asks for Input Monitoring, never Accessibility. In the callback, immediately discard the CGEvent's key code and character payload after incrementing an in-memory counter: do not log, store, or transmit the keycode, unicode string, or modifier-key identity, only a running integer count and, for mouse, a running float or double of accumulated pixel distance. Re-emit the event unmodified (return Unmanaged.passUnretained(event)) since the tap must not alter or block user input. Handle the CGEventTapDisable callback (a tap can be silently disabled by the system, e.g. under load) by re-enabling it, and re-check permission status on app foreground in case the user revoked it in System Settings. Distribute both inside and outside the Mac App Store if possible: ship the sandboxed build to the Store (accepting the residual review-outcome risk noted above) and a notarized direct-download build as the primary or fallback channel, mirroring how most utilities in this space (WhatPulse, BetterTouchTool, CleanShot X) actually ship.

**Fallback.** If Input Monitoring is denied or revoked, or if Mac App Store review rejects the CGEventTap-based build, fall back to NSEvent.addGlobalMonitorForEvents(matching:) for keyDown/mouseMoved/etc., which is documented as gated by Accessibility (AXIsProcessTrusted) rather than Input Monitoring. This still only observes events (it cannot modify or swallow them, satisfying the listen-only design constraint) but requires the broader Accessibility grant, is more likely to draw App Review scrutiny, and per one forum report can silently register successfully while never firing callbacks if Accessibility is not actually granted, so the app must independently poll AXIsProcessTrusted() and surface a clear permission-needed state to the user rather than assuming a monitor object's existence means it is working. As a last resort with no permission granted at all, the app should degrade gracefully by showing all other Byte metrics (AI usage, network, disk, screen time) with the input-tracking tile visibly disabled and a one-click deep link to the relevant System Settings pane, rather than failing to launch or blocking the rest of the dashboard.

**Permissions.**
- Input Monitoring (TCC service kTCCServiceListenEvent, shown in System Settings > Privacy & Security > Input Monitoring) -- required to create a listen-only global CGEventTap or to pass IOHIDCheckAccess/IOHIDRequestAccess for kIOHIDRequestTypeListenEvent. This is the permission a count-only, never-record-which-key implementation should target.
- Accessibility (AXIsProcessTrusted, TCC service kTCCServiceAccessibility) -- required only if using NSEvent.addGlobalMonitorForEvents or a non-listenOnly (.defaultTap) CGEventTap that can also modify/block events. Not needed for a pure counting use case and should be avoided since it is a broader, more sensitive permission (also grants UI automation of other apps) that is a known red flag for sandboxed/App-Store apps. Multiple sources note that if Accessibility is already granted, Input Monitoring access is implicitly included, but not vice versa.
- App Sandbox entitlement: no dedicated 'input monitoring' entitlement key exists in the App Sandbox entitlement list; access is governed purely by the runtime TCC prompt/consent, not a com.apple.security.* entitlement. com.apple.security.device.usb / raw HID entitlements do NOT grant this and are the wrong path (confirmed TCC-denied even with the entitlement present).
- Standard code-signing / notarization requirements apply; TCC grants are tied to the app's code signature identity, and re-signing (e.g. ad-hoc rebuilds during development, or CI re-signing) can silently invalidate a previously granted Input Monitoring approval, requiring the user to re-grant it.

**App Store compatibility.** Partial: technically supported (a listen-only CGEventTap gated by Input Monitoring works inside the App Sandbox, per an Apple DTS engineer's explicit recommendation and per several shipped Mac App Store apps such as KeyCount, Keystroke Counter, Key Click Pro, and Keystroke Pro), but a direct 2020 developer forum question asking Apple to confirm this exact pattern for the App Store went unanswered, and App Review guideline 2.4.5(v) language about apps that request keystroke access has caused at least one unrelated app's rejection, so review outcomes for this category are not guaranteed and should be treated as a real, non-zero risk rather than a settled yes.

**Energy cost.** No official Apple benchmark was found. Reasoned estimate: a listen-only CGEventTap's callback runs synchronously, in-process, on the same thread as the tap's run loop, once per keyboard or mouse event system-wide. Because the callback here only needs to increment an integer counter or accumulate a float delta (no key-code inspection, no IPC, no disk write per event), the marginal CPU cost per event is on the order of microseconds, and aggregate CPU usage under normal human typing and mouse-movement rates (tens of events per second at most, versus scroll-wheel or high-frequency mouseMoved streams which can run much higher, potentially hundreds of events per second during a fast mouse swipe) should stay well under one percent CPU on any Mac from the last decade. The larger energy consideration is idle cost: the tap's run loop must stay alive and awake for the process's entire lifetime, which shows up in Activity Monitor's Energy tab as always-on background activity typical of any resident menubar helper, a standing but small idle energy tax rather than a per-event cost. Batching counter writes to disk (rather than writing on every event) is recommended to keep disk I/O energy cost negligible relative to the tap itself.

**Candidate APIs.**
- `CGEvent.tapCreate (CGEventTapCreate) with options: .listenOnly` (public framework): Core Graphics / Quartz Event Services (import CoreGraphics / ApplicationServices). Creates a global event tap (place: .cgSessionEventTap or .cgHIDEventTap) that observes keyDown, keyUp, flagsChanged, mouseMoved, leftMouseDragged, scrollWheel, etc. across all processes. Passing options: .listenOnly (rather than .defaultTap) makes the tap read-only and gates it behind the Input Monitoring TCC permission instead of Accessibility. An Apple DTS engineer (Quinn) explicitly recommended this API on the Apple Developer Forums to a developer building a sandboxed app that wanted to count keypresses, in preference to HID device access which is blocked by the sandbox (https://developer.apple.com/forums/thread/724608). A separate 2020 forum question asking Apple to confirm App Store approval of this exact pattern for a text-expansion utility went unanswered (https://developer.apple.com/forums/thread/668975). For mouse distance, sum the delta fields (or endpoint coordinates) of mouseMoved/dragged events from the same tap.
- `CGPreflightListenEventAccess() / CGRequestListenEventAccess()` (public framework): Quartz Event Services helpers for checking/prompting for the Input Monitoring privilege specifically for listen-only event taps, as distinct from the Accessibility-gated CGRequestPostEventAccess. Preflight is a non-blocking check; Request triggers the system consent prompt and, once denied, requires the user to flip the toggle in System Settings manually (no re-prompt).
- `IOHIDCheckAccess(kIOHIDRequestTypeListenEvent) / IOHIDRequestAccess(kIOHIDRequestTypeListenEvent)` (public framework): IOKit HID System (IOKit/hidsystem/IOHIDLib.h). Lower-level equivalent of the CG preflight/request calls; returns/queries an IOHIDAccessType (kIOHIDAccessTypeGranted/Denied/Unknown) for the same Input Monitoring privilege. Used by several real-world apps (e.g. Scroll Reverser, OpenEmu) to check Input Monitoring status before creating an event tap or opening HID devices. This is the same underlying TCC service CGEventTap's listenOnly option checks; it is not a separate or additional permission from Input Monitoring shown in System Settings.
- `NSEvent.addGlobalMonitorForEvents(matching:handler:)` (public framework): AppKit API for global (read-only, cross-app) event monitoring. Per multiple sources this is gated by the Accessibility permission (AXIsProcessTrusted), not Input Monitoring -- it silently returns a monitor object but never fires callbacks if Accessibility is not granted. Accessibility is a broader, more sensitive permission than Input Monitoring (it also grants UI scripting/control of other apps) and is understood by the community and App Review commentary to be a red flag for sandboxed App Store apps. For a count-only use case, CGEventTap+listenOnly (gated by the narrower Input Monitoring permission) is the better-fitting and more sandbox-friendly API.
- `IOHIDManager / IOHIDDeviceOpen (raw HID device access)` (public framework): Confirmed NOT usable for this purpose in a sandboxed app: a developer got a 'TCC deny IOHIDDeviceOpen' error even with the com.apple.security.device.usb entitlement enabled, and Apple DTS explicitly redirected them to CGEventTap instead (https://developer.apple.com/forums/thread/724608). Documented here as a dead end to avoid.
- `WhatPulse (real-world precedent, not an API to call)` (third-party service): WhatPulse's own documentation states it uses Accessibility to 'monitor keyboard and mouse events' and Input Monitoring to 'track which applications you're using,' and that it counts keystrokes 'in an accumulated manner rather than logging the order in which keys are typed' -- i.e. count-only, no key identity retained. WhatPulse is distributed only via direct download and Homebrew (whatpulse.org/downloads), not the Mac App Store; its docs make no mention of sandboxing. Several smaller, similar apps (KeyCount, Keystroke Counter, Key Click Pro, KeyLimePie Keystrokes, Keystroke Pro) are on the Mac App Store today; KeyCount's README confirms it requests only Input Monitoring, not Accessibility.

**Caveats.**
- Secure Input mode (activated automatically by password fields, and by apps like Terminal's Secure Keyboard Entry) makes keystrokes in that context invisible to any event tap, including listen-only ones. This is an intentional macOS security feature (documented in the old Apple TN2150), not a bug, but it means keystroke counts will systematically undercount actual typing whenever the user is in a password field or a secure-input app.
- The Input Monitoring versus Accessibility permission boundary is not perfectly clean: multiple sources note that having Accessibility already granted silently also grants Input Monitoring, and one forum thread describes inconsistent behavior in early Catalina betas where some event-tap apps did not trigger the Input Monitoring prompt at all for unclear reasons. Exact behavior may vary slightly by macOS version.
- TCC grants for Input Monitoring are tied to code-signing identity; re-signing the app binary (common during development or with certain CI/notarization pipelines) can silently invalidate a previously granted permission without any user-visible error, causing the tap to appear created successfully but never deliver events. This was documented independently by a developer in a February 2026 blog post as a 'silent disable race.'
- A CGEventTap can be disabled by the system at any time (e.g. if its callback is judged too slow) via a kCGEventTapDisabledByTimeout callback; a robust implementation must detect this and call CGEvent.tapEnable to restore it, or silently lose all counting from that point forward.
- Mouse travel distance is not a native macOS metric; it must be derived by the app itself from summing per-event pixel deltas across mouseMoved/mouseDragged events, and its accuracy depends on the tap not dropping events, on multi-monitor coordinate handling, and on trackpad versus mouse deltas being on comparable scales. There is no Apple-provided authoritative counter to cross-check against.
- No authoritative, quantified Apple source or independent benchmark was found for the specific CPU/energy cost of a permanent listen-only CGEventTap; the energy_cost figure below is a reasoned estimate based on the tap's callback being invoked synchronously on every keyboard/mouse event system-wide, not a measured citation.
- Whether the Mac App Store will actually approve a given ByteLife build using this technique is unresolved in the public record: the one forum thread asking this exact question outright (thread 668975) received zero replies, and App Review guideline 2.4.5(v) has been invoked (apparently in error, traced to a third-party SDL library bug in one reported case) against at least one unrelated app, so review risk should be treated as real even though multiple similar apps are live on the Store today.
